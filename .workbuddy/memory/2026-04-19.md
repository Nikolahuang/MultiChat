## 关键修复：WebView 预览时界面不消失
**根因**：App.tsx 中 `previewProvider ? <黑底浮动标签> : <SyncMode />` —— 打开 WebView 时整个 SyncMode 被替换掉了，Tab 栏、历史侧栏全没了
**修复**：
1. App.tsx 移除 previewProvider 条件替换逻辑，始终渲染 SyncMode
2. SyncMode 内将浮动标签改为固定信息栏（Tab 栏下方），显示当前对话平台 + 状态（回复中/已回复）+ 关闭按钮
3. 固定信息栏不会被 WebView 遮挡或消失

## 关键修复：WebView 遮挡 Tab 栏问题
**根因**：Rust 端 `show_provider` 用 `y=0` 定位 WebView，从窗口最顶部开始覆盖
**修复方案**：
1. AppState 新增 `webview_top_offset` 字段（默认 88px）
2. `show_provider` / `create_webview` / resize 处理全部使用 top_offset：
   - y 坐标 = sidebar_width（x）, top_offset（y）
   - 高度 = 窗口高度 - top_offset
3. 新增 `set_webview_top_offset` Rust command
4. 前端 SyncMode 用 `ResizeObserver` + ref 测量 Tab 栏+信息栏的实际物理像素高度，动态通知 Rust 端
5. 这样 WebView 只在内容区显示，不遮挡标题栏和 Tab 栏

## UI 重构：WebView 预览紧凑模式
**问题**：
1. 打开 WebView 后，React 侧仍然渲染完整布局（历史侧栏+空白区），大片暗色空白浪费空间
2. ResizeObserver 只监听 header 区域，窗口 resize 时可能没触发重新测量

**修复 — 紧凑预览模式**：
- **打开 WebView 时自动隐藏**：历史消息侧栏（`w-0 overflow-hidden`）、InputBar
- **Tab 栏精简**：隐藏历史切换按钮、清空按钮等不需要的操作
- **信息栏压缩为单行**：`🔗 当前对话 · DeepSeek · ✓已回复 | 复制 ✕关闭`
- **内容区域只渲染一个纯黑 div 占位**：让 WebView 有完整的空间显示
- **ResizeObserver + window.resize 双重监听**：确保窗口调整时始终重新计算偏移量
- **previewProvider 变化时触发重测**：依赖项中加入 previewProvider

**效果对比**：
```
修复前: ┌─ Tab 栏 ─────────────┐
        ├─ 信息栏 ─────────────┤  ← React 渲染的完整 UI
        ├─ (历史侧栏 w-64) ────┤     浪费大量空间
        ├─ 暗色空白区 ──────────┤
        ├─ InputBar ────────────┤
        └───────────────────────┘

修复后: ┌─ 精简Tab栏(无历史按钮) ─┐
         ├─ 单行信息栏(紧凑) ──────┤  ← 仅这两行是 React 的
         ├─ WebView 全屏展示 ──────┤  ← AI 对话界面占满剩余空间
         └────────────────────────┘
```

## 性能优化：切换卡顿和资源消耗高
**根因**：频繁 IPC 调用 + 无 debounce + 重复 WebView 操作

**前端优化**：
1. `measureAndNotify` 加 **150ms debounce**——拖动窗口时不再每帧触发 invoke
2. **值缓存跳过**：offset 变化 < 2px 时直接 return，不发送 IPC
3. `scrollIntoView({ behavior: 'smooth' })` → `scrollIntoView(false)`（instant，避免 layout thrashing）

**Rust 优化**：
1. `show_provider` 加 **`last_shown_provider` 缓存**：重复点击同一 Tab 直接 return Ok(())
2. `set_webview_top_offset` 跳过相同值（差 < 2px 不更新）
3. resize handler 中 **合并 3 次 lock 为 1 次读取块**（减少锁竞争）
4. 注释掉调试 eprintln（高频调用时 I/O 开销也不可忽略）

## 关键修复：窗口拖拽调整大小时 WebView 出现空边
**问题**：拖拽窗口边缘调整大小时，WebView 中间出现空边，无法自适应跟随窗口变化

**根因分析（3 个问题）**：
1. **`sidebar_width` 永远是硬编码的 280.0 物理像素**：前端从未调用 `set_sidebar_width` 更新，实际侧边栏宽度是 `w-64`(256px) 或折叠时 `w-11`(44px)，导致 WebView x 坐标和宽度计算错误
2. **Resize 事件处理器只更新了 WebView 的 SIZE（宽高），没有更新 POSITION（x, y）**：所以窗口缩放后 WebView 位置不变，出现错位和空边
3. **前端测量 top_offset 时硬编码了 `titleBarHeight = 36`**：这个值可能不准确，且在不同 DPI 下会有偏差

**修复方案**：

前端 (`SyncMode.tsx`)：
1. **ResizeObserver 同时测量两样东西**：
   - headerAreaRef 高度 → `set_webview_top_offset`（去掉了硬编码 titleBarHeight=36）
   - **侧边栏实际宽度** → 新增调用 `set_sidebar_width`（通过 DOM 查询 bg-[#0d0d15] + border-r 的元素获取真实宽度）
2. debounce 从 150ms 减到 100ms（resize 时响应更快）
3. 侧边栏选择器用遍历方式匹配 class 字符串（更可靠）

Rust 端 (`lib.rs`)：
1. **Resize handler 同时更新位置 AND 尺寸**：
   ```
   set_position(PhysicalPosition::new(new_x, new_y))  // ← 新增
   set_size(PhysicalSize::new(new_w, new_h))
   ```
2. debounce 从 300ms 减到 150ms（跟随更快）
3. 类型修复：`.max(100)` → `.max(100.0)`（f64 类型要求）

`commands.rs`：
- `set_sidebar_width` 加入变化检测：差 < 1px 不更新

## 群聊界面重构
**问题**：群聊功能不符合常规群聊界面和形式
**根因**：
1. 右侧成员面板占太多空间，不像微信群聊
2. 有一个不合理的 "WebView模式" 切换按钮（群聊应该纯文字交流）
3. streaming消息合并逻辑有bug：直接push可能重复、thinking状态残留
4. 发送消息时插入不必要的系统提示消息"消息已发送给 X 位成员"
5. 消息没有时间分组

**修复**：
1. **去掉右侧成员面板**，改为顶栏紧凑头像展示 + 可折叠成员栏
2. **去掉WebView模式切换**，改为点击AI头像/成员按钮直接打开WebView
3. **用 useMemo 重写消息合并逻辑**：用 Set 追踪已处理的 streaming 消息，避免重复；按 timestamp 排序
4. **去掉系统提示消息**，发送后不再插入"消息已发送给 X 位成员"
5. **添加时间分组**：超过5分钟间隔显示时间分割线
6. **消息气泡hover显示操作按钮**：复制 + 打开WebView
7. **用户消息右对齐蓝色气泡**，AI消息左对齐带头像+昵称（类微信布局）
